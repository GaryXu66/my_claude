# 生成测试代码

## 适用场景

- 为 Service 层生成单元测试
- 为 Controller 层生成单元测试
- 为完整流程生成集成测试
- 为非功能需求生成测试（性能、安全、兼容性）
- 提高测试覆盖率

---

## 测试类型选择

| 类型 | 场景 | 特点 |
|------|------|------|
| 单元测试 | 测试单个方法逻辑 | 快速、Mock 依赖 |
| 集成测试 | 测试完整流程 | 真实数据库、慢 |
| 性能测试 | 验证响应时间、并发能力 | JMeter、Gatling |
| 安全测试 | 验证参数校验、注入防护 | OWASP ZAP、手动测试 |

**建议：** 核心业务逻辑用单元测试，关键流程用集成测试，对外接口必须包含安全测试，有性能要求时必须生成性能测试。

---

## 测试框架

| 框架 | 用途 |
|------|------|
| JUnit 5 | 测试框架 |
| Mockito | Mock 框架（模拟对象和方法调用） |
| PowerMock | Mock 静态方法、私有方法、构造函数 |
| AssertJ | 断言库 |
| SpringBootTest | 集成测试 |
| JMeter | 性能测试 |

## 测试原则

1. **单一职责** - 每个测试只测一个场景
2. **独立性** - 测试之间互不依赖
3. **可重复** - 多次运行结果一致
4. **自验证** - 测试自动判断通过/失败
5. **及时性** - 与代码同步编写
6. **全面性** - 功能测试与非功能测试并重

## 单元测试规范

### 测试范围

- **必须测试**：Service 类、核心业务逻辑、工具类
- **必须覆盖**：参数边界测试、空指针、数组越界等低级错误
- **无需测试**：接入层、RPC、DAO、HTTP 等外部调用（使用 Mock/Stub 代替）
- **DAO 层测试**：如需测试，注意数据回滚，避免产生脏数据

### 测试原则

- 单元测试应保持独立可运行，不依赖运行环境
- 每个测试方法只测试一个功能点
- 测试之间互不依赖，可任意顺序执行

### 单元测试执行规范

- **Bug 修复时** - 必须增加单元测试覆盖修复的场景
- **重构时** - 被依赖的业务模块建议有单元测试保护
- **开发优先级** - Service 业务模块层优先写单元测试
- **代码审查** - 代码审查时必须检查单元测试

### 测试结构

采用 Given-When-Then 模式：
- **Given** - 准备数据（Mock 依赖、设置预期值）
- **When** - 执行操作（调用被测方法）
- **Then** - 验证结果（断言返回值、验证 Mock 调用）

### Mock 使用规范

对于业务场景涉及 DAO 请求、RPC、HTTP 第三方调用等，可以模拟对应请求的返回值而不真正执行调用逻辑，快速测试业务逻辑是否符合预期。

#### Mockito 注解

| 注解 | 说明 |
|------|------|
| `@InjectMocks` | 创建实例，其余用 `@Mock` 或 `@Spy` 注解创建的 mock 将被注入到该实例中 |
| `@Mock` | 对函数的调用均执行 mock（虚假函数），不执行真正部分 |
| `@Spy` | 对函数的调用均执行真正部分 |

#### PowerMock 注解

| 注解 | 说明 |
|------|------|
| `@RunWith(PowerMockRunner.class)` | 表明使用 PowerMockRunner 进行测试 |
| `@PrepareForTest({})` | 把需要用 PowerMock 进行特殊 mock 操作的类放入该注解，包括 final 类、私有方法、静态方法等 |
| `@SuppressStaticInitializationFor("")` | 阻止对应类的静态代码块执行 |
| `@PowerMockIgnore()` | 表明不使用 PowerMock 加载所声明的 package 路径的类，用于解决系统类异常 |

#### Mockito 使用场景

| 场景 | 说明 |
|------|------|
| 模拟对象和方法调用 | Mock 外部依赖（RPC、数据库、HTTP） |
| 验证方法调用 | 使用 `verify` 验证方法是否被调用 |

#### PowerMock 使用场景

| 场景 | 说明 |
|------|------|
| 静态方法 | Mockito 不支持，使用 PowerMockito.mockStatic() |
| 私有方法 | 使用 PowerMockito.when(obj, "methodName") |
| 构造函数 | 模拟对象创建时的返回值 |
| final 方法 | 需要配合 @PrepareForTest 使用 |

### 断言规范

优先使用 AssertJ，支持链式调用。

## 集成测试规范

### 使用场景

- 测试多个组件协作
- 测试数据库操作
- 测试事务行为
- 测试 Spring 配置

### 基本要求

- 使用 `@SpringBootTest` 启动 Spring 容器
- 使用 `@Transactional` 确保测试后数据回滚
- 使用 `@ActiveProfiles("test")` 隔离测试环境
- 使用测试数据库，与生产环境隔离

## 非功能需求测试

### 检测国际化配置

生成测试前，先检测项目是否包含国际化功能：

**检测条件（满足任一即认为有国际化）：**
1. `src/main/resources/i18n/` 目录存在
2. `application.yml` 中配置了 `spring.messages.basename`
3. 项目中存在 `MessageSource` 相关配置类
4. 代码中使用了 `I18nUtils` 或 `MessageSource`

**检测方式：**
```bash
# 检查 i18n 目录
ls src/main/resources/i18n/

# 检查配置文件
grep -r "spring.messages" src/main/resources/

# 检查代码引用
grep -r "MessageSource\|I18nUtils\|LocaleContextHolder" src/main/java/
```

### 测试内容

| 类型 | 检测条件 | 测试内容 | 执行方式 | 参考模板 |
|------|----------|----------|----------|----------|
| 性能测试 | 有性能要求 | 响应时间、并发、压力测试 | 自动化 | [性能测试模板](templates/performance-test.md) |
| 安全测试 | 对外接口 | SQL注入、XSS、参数校验、频率限制 | 自动化/手动 | [安全测试模板](templates/security-test.md) |
| 兼容性测试 | 对外接口 | RESTful规范、JSON格式、HTTP状态码 | 自动化/手动 | [测试报告模板](templates/test-report.md) |
| 国际化测试 | 检测到国际化配置 | 多语言消息、Accept-Language请求头 | 自动化 | [国际化测试模板](templates/i18n-test.md) |

### 执行方式说明

- **自动化**：通过 JUnit、JMeter 等工具自动执行
- **手动**：需要人工验证，如 CSRF、越权、RESTful 规范检查
- **混合**：部分场景可自动化，部分需要手动验证

### 验证标准

- 单次请求响应时间 < 100ms
- 并发100用户时错误率 < 1%
- 参数校验失败返回明确错误提示
- 服务异常时优雅降级

### 测试报告要求

**重要：**
- 报告中只记录已实际执行的测试项及其结果
- 不包含未执行的测试项
- 必须标注每项测试的执行方式（自动化/手动）
- 测试内容必须与需求文档的内容一一对应

### 测试报告模板使用

生成测试并运行完成后，使用 [测试报告模板](templates/test-report.md) 生成正式测试报告：

1. **运行测试**：执行 `gradle test` 或 `mvn test` 获取实际测试结果
2. **收集数据**：从测试输出中获取统计数据（总数、通过、失败、覆盖率）
3. **填写报告**：按照模板结构填写，注意：
   - 只记录**已实际执行**的测试项
   - 测试内容必须与**需求文档**一一对应
   - 标注每项的**执行方式**（自动化/手动）
   - 不得编造未执行的测试结果

报告结构：
- **单元测试报告**：记录 Service 层、Controller 层、工具类的测试覆盖情况
- **集成测试报告**：记录完整流程测试场景，与需求一一对应
- **非功能测试报告**：记录性能、安全、兼容性测试结果
- **结论**：汇总通过/不通过项、风险评估、改进建议

## 注意事项

1. **Mock 外部依赖** - 数据库、Redis、第三方接口
2. **测试边界条件** - null、空集合、边界值
3. **测试异常场景** - 业务异常、系统异常
4. **保持测试简洁** - 避免测试代码过于复杂
5. **安全优先** - 对外接口必须包含安全测试
6. **性能基线** - 建立性能基线，便于回归对比
7. **国际化检测** - 检测到国际化配置时自动生成国际化测试用例
