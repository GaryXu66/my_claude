# proj-analyze-req

> **核心目标**：通过结构化的需求澄清，准确理解用户真实需求，明确功能边界和业务逻辑
> **关键产出**：需求分析文档 + 全流程任务文档骨架
> **下一步**：用户确认后进入技术方案设计（/proj-analyze-design）

---

## 执行流程

```
Step 1: 初始化任务文档 → Step 2: 需求理解 → Step 3: 需求澄清 → Step 4: 生成文档 → Step 5: 用户确认
```

---

## Step 1: 初始化任务文档

用户输入需求后，**必须立即**创建全流程任务文档骨架。

**文件位置**：`docs/task/{YYYYMMDD}_{中文模块名}_任务.md`

**内容要求**：
- 使用任务文档模板（见 `/proj-task`）
- 仅填写：流程状态、产物清单、上下文快照、下一步指令
- **禁止填写**：技术实现细节、字段设计、具体方案

---

## Step 2: 需求理解

### 2.1 初步提取

从用户描述中提取核心信息：

| 维度 | 提取内容 |
|------|----------|
| 功能目标 | 要实现什么功能 |
| 使用角色 | 谁来使用 |
| 涉及端 | Web后台 / 小程序 / App / 开放接口 |
| 涉及模块 | 影响哪些现有模块 |
| 关键操作 | 核心功能点 |

### 2.2 规模判断

| 规模 | 特征 | 处理策略 |
|------|------|----------|
| **简单** | 改字段、加接口、改配置 | 可简化提问，但仍需确认 |
| **中等** | 新模块、独立功能 | 标准澄清流程 |
| **复杂** | 跨模块、架构调整、核心流程 | 完整澄清，详细文档 |

**判断依据**：
- 涉及 1 个文件 → 简单
- 涉及 1 个模块（多文件） → 中等
- 涉及多个模块 → 复杂

---

## Step 3: 需求澄清（核心）

根据需求类型，向用户澄清以下维度：

| 维度 | 核心问题 | 典型示例 |
|------|----------|----------|
| **功能范围** | 谁用？哪些端？ | Web后台管理端？小程序端？还是都需要？ |
| **用户角色** | 哪些角色？各角色权限？ | 管理员、普通用户、审核员？各自能做什么？ |
| **功能清单** | 具体功能点？ | 列表查询、详情查看、新增、编辑、删除、导出、审批？ |
| **需求优先级** | 功能优先级？研发排期？ | 哪些是P0必须做的？哪些是P1重要的？哪些是P2可延后的？ |
| **业务流程** | 主流程？状态流转？ | 提交→审核→批准→执行？是否有驳回、撤回？ |
| **异常场景** | 异常情况如何处理？ | 提交失败怎么办？审批驳回后如何修改？超时如何处理？ |
| **业务规则** | 限制条件？校验规则？ | 最多提交几次？谁可以审批？数据权限范围？ |
| **数据要求** | 记录哪些信息？ | 标题、内容、附件、状态、创建时间？哪些必填？ |
| **非功能性需求** | 性能？安全？兼容性？ | 并发量要求？响应时间要求？兼容哪些浏览器/系统版本？ |
| **关联关系** | 关联哪些模块？ | 关联用户？关联订单？关联审批流程？ |
| **参考资料** | 原型图？参考产品？ | 有没有设计稿、参考链接？ |

## Step 4: 生成需求文档

### 4.1 文档模板

使用需求文档模板：`templates/req-doc.md`

**保存位置**：`docs/req/{YYYYMMDD}_{中文模块名}_需求.md`

### 4.2 关键章节

| 章节 | 内容要点 | 输出要求 |
|------|----------|----------|
| **1. 功能概述** | 业务背景、功能定位、核心价值 | 简洁描述，不超过100字 |
| **2. 用户角色与场景** | 角色定义、典型使用场景 | 每个主要功能都要有场景描述 |
| **3. 功能需求详述** | 具体功能点、输入输出、业务规则 | 详细描述，包含前置条件、输入、输出、规则 |
| **4. 业务流程** | 主要业务流程、状态流转 | 简单流程用文字（≤3步），复杂流程用Mermaid图（>3步） |
| **5. 数据需求** | 业务视角的数据实体、约束规则 | 描述需要记录什么信息，不涉及表结构 |
| **6. 非功能性需求** | 性能、安全、兼容性 | 并发量、响应时间、权限控制等 |
| **7. 约束条件** | 技术约束、业务约束 | 消息通知方式、数据实时性、保留策略等 |
| **8. 验收标准** | 功能、性能、安全验收 | 可执行的验收条件 |
| **9. 风险评估** | 技术风险、业务风险 | 风险等级和应对措施 |

### 4.3 文档边界

**✅ 需求文档包含**：
- 业务背景和目标
- 用户角色和使用场景
- 功能需求详述（做什么）
- 业务流程和规则
- 数据需求（业务视角）
- 非功能性需求
- 验收标准

**❌ 需求文档禁止包含**：
- 数据库表设计（字段名、类型、索引）
- 接口定义（URL、参数格式、响应结构）
- 技术架构设计
- 代码结构设计
- 具体技术选型

### 4.4 流程图规范

- **简单流程（≤3步）**：用文字描述
  ```
  步骤1 → 步骤2 → 步骤3
  ```

- **复杂流程（>3步或有分支）**：必须使用Mermaid流程图
  - **普通流程图**：适用于单一角色或不需要区分角色的流程
  ```mermaid
  graph TD
      A[流程开始] --> B{判断条件}
      B -->|条件1| C[操作1]
      B -->|条件2| D[操作2]
      C --> E[后续步骤]
      D --> E
      E --> F[流程结束]
  ```

  - **泳道图**：适用于多角色协作的复杂业务流程，清晰区分各角色操作
  ```mermaid
  graph TD
      subgraph 用户
          A[提交申请] --> B{等待审核}
      end

      subgraph 审核员
          B -->|通过| C[批准]
          B -->|驳回| D[修改意见]
      end

      subgraph 用户
          D --> E[修改重新提交]
          E --> B
      end

      subgraph 审核员
          C --> F[归档]
      end
  ```

### 4.5 同步任务文档

更新任务文档：
- 标记"需求分析"状态：`[ ]` → `[~]` → `[x]`
- 更新"产物清单"：记录需求文档路径
- 更新"上下文快照"：填写功能名称、端类型、关键约束

---

## Step 5: 用户确认

### 5.1 确认话术

输出需求文档后，**必须**使用以下话术：

```
以上是我对需求的理解，请确认：
1. 功能范围是否完整？
2. 业务流程是否正确？
3. 是否有遗漏的功能点或业务规则？

确认后我将进入技术方案设计阶段（/proj-analyze-design）。
```

### 5.2 反馈处理

| 用户反馈 | 处理方式 |
|----------|----------|
| "确认" / "没问题" / "可以" | 进入阶段二：`/proj-analyze-design` |
| "需要补充 xxx" | 更新需求文档，再次确认 |
| "需要修改 xxx" | 修改对应内容，再次确认 |
| "取消" / "不做了" | 结束流程 |

### 5.3 确认后更新

更新任务文档：
- 标记"需求确认"状态：`[ ]` → `[x]`
- 更新"下一步指令"：进入技术方案设计

---

## 重要注意事项

- **一次一个问题** — 不要用多个问题让人应接不暇
- **优先选择选择题** — 在可能的情况下，选择题比开放式问题更容易回答
- **果断践行 “YAGNI” 原则** — 对照用户需求，移除不必要的功能
- **保持灵活性** — 当某些内容不合理时，回过头去澄清
- **禁止合并阶段** - 需求确认前不要开始技术方案
- **严格边界** - 需求文档不能包含技术实现细节
- **持续迭代** - 复杂需求可能需要多轮澄清，过程中需求有变化，及时更新文档
