# 后端异常处理

## 何时抛出异常

无法完成正常流程应抛出异常，而不是返回一个null或错误码来表示没有完成任务。

```java
// 正确的例子
@NotNull
User getUserById(long id) {
    // 找不到此id就抛出异常，而不是返回null. 避免调用者每次都去检查返回值。
}
```

除非是流程中显而易见的分支，例如 string 转 int 有明确的成功和失败，或数据访问层的 tryGetUserById，应在方法名上明确使用 try/ find 单词前缀。

```java
// 正确的例子： 数据访问层，找不到此数据是一个期望的路线
@Nullable
User tryGetUserById(long id) {
    // 找不到此id 返回null，
    // 但是诸如数据库连接不上，不要拦截异常返回null，应如实抛出。
}
```

## 异常分类

**商业异常**：凡是期望让用户看到，对用户有价值，用户可处理的异常，使用 BusinessException。例如权限不足，单据不存在，库存不足。

**系统异常**：凡是因为软件BUG,系统级故障造成流程的中断，视为系统异常。这类异常记录服务端日志，仅报告用户“当前操作出现问题，未能正确完成，请联系 xxx@yyy.com"。
例如 null指针,数据库无法连接,内存溢出。

## 如何抛出异常

- 选择合适的异常类型
- 异常消息 [应使用 i18n 获取本地化消息](i18n.md)

```java
throw new BusinessException(i18n.userErrorNotFound(), "id", id.toString());
```

## 异常的全局自动处理

软件内建全局的异常拦截，保证：
- 在系统的边界不会因为异常导致系统崩溃
- 使用白名单等过滤机制，区分商业异常和系统异常
- 记录系统异常到日志，在可选的情况下将商业异常记录（默认关闭）
- 商业异常尽可能忠实反馈给客户端，不记录日志（默认）

## 安全考虑

- 生产环境中，商业异常不创建栈(writableStackTrace=false)
- 非商业异常记录日志，但不将具体原因告知用户
- 过滤敏感信息，BusinessException 中 arguments 集合中key以 _(下划线) 开头的参数不能序列化到客户端

## 高级功能

**监控指标**: 全局统一处理，异常的频率和类型

**可追溯性**：全局统一处理，使用Result 的 traceId 跟踪异常的链路
